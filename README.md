# Syria Campaign · Donation Anomaly Detection

A compliance tool for detecting anomalous or potentially bad-actor donations in a humanitarian campaign context. Adapted from a bank transaction fraud dataset, extended with domain-specific feature engineering, realistic synthetic labels, and a Streamlit dashboard for risk triage.

> **Dataset note:** The original Kaggle `Is_Fraud` labels are randomly assigned and carry zero predictive signal. `generate_labels.py` replaces them with rule-based compliance labels. See [The Label Problem](#️-the-label-problem) below.

## Overview

This system was built to detect suspicious donation patterns for the Syria Campaign — flagging potential money laundering via charity, structuring, pass-through accounts, and other compliance risks. It uses the [LOL Bank Pvt. Ltd. Kaggle dataset](https://www.kaggle.com/datasets/marusagar/bank-transaction-fraud-detection) as its data source, remapped to a donation domain vocabulary.

The pipeline is three scripts run in order:

| Script | What it does | When to run |
|---|---|---|
| `generate_labels.py` | Replaces random fraud labels with compliance rules | Once, before training |
| `prepare.py` | Trains models, writes `artifacts/` to disk | Once, or when retraining |
| `app.py` | Streamlit dashboard | Any time |

## Quickstart

**Requirements:** Python 3.11+, [`uv`](https://github.com/astral-sh/uv)

```bash
# Install dependencies
uv add pandas numpy scipy scikit-learn joblib pyarrow streamlit matplotlib

# Step 1 — generate realistic fraud labels (~10 seconds)
uv run python generate_labels.py

# Step 2 — train models and cache artifacts (~2–4 minutes, runs once locally)
uv run python prepare.py

# Step 3 — launch the dashboard
uv run streamlit run app.py
```

> `prepare.py` is the only heavy step. It runs once on your machine and writes compressed joblib + parquet artifacts to `artifacts/`. The Streamlit app loads these instantly — no training happens in the browser.

## ⚠️ The Label Problem

After the initial pipeline was built and the Precision-Recall curves were plotted, all three models performed at essentially random chance — Average Precision ~0.052, ROC-AUC ~0.50.

**Root cause:** The `Is_Fraud` column in the original Kaggle dataset is randomly assigned. It encodes no real-world patterns whatsoever. Every diagnostic confirmed this:

| Signal check | Result |
|---|---|
| Correlation: `Transaction_Amount` vs `Is_Fraud` | −0.0021 |
| Correlation: `Age` vs `Is_Fraud` | −0.0015 |
| Correlation: `Account_Balance` vs `Is_Fraud` | +0.0001 |
| Fraud rate across all `Transaction_Type` values | 4.93% – 5.19% (completely flat) |
| Fraud rate across all `Device_Type` values | 4.99% – 5.10% (completely flat) |
| Fraud rate across all `Merchant_Category` values | 4.82% – 5.20% (completely flat) |
| Fraud rate variance across 34 states | std dev 0.0027 (noise-level) |
| Top 1% largest transactions | 5.85% vs 5.04% baseline (negligible) |

This is a Kaggle dataset designed for *practising the ML pipeline* — data loading, feature engineering, model building — not one where labels were generated from feature values. No model, however sophisticated, can learn from noise.

**Fix:** `generate_labels.py` rebuilds the labels from scratch using 12 weighted compliance rules. See below.

## Synthetic Label Generation

Labels are generated in three stages:

1. **Rule scoring** — each row is scored 0–N based on how many weighted compliance rules it triggers
2. **Noise injection** — Gaussian noise (σ=0.12) blurs hard boundaries so models learn soft probabilities, not memorisable cutoffs
3. **Threshold calibration** — a percentile threshold produces a realistic ~5.5% anomaly rate

### Compliance rules

| Rule | Threat modelled |
|---|---|
| Night (00:00–04:59) + Transfer + large (≥p75) | Automated / scripted donations |
| High-risk platform + large amount | Untraceable channels (Virtual Card, QR Code, Chatbot) |
| Donation / Balance ratio > p90 | Pass-through account |
| Low balance (< p10) + large gift (≥p75) | Mule account |
| Large round-number amount (mod 1000 = 0) | Structuring / money laundering |
| Just below 50k / 75k / 90k / 99k (within 1%) | Threshold avoidance |
| Very large withdrawal (≥p95) | Layering |
| Business account + night + transfer | Corporate money laundering via charity |
| Weekend + night + transfer | Coordinated / off-hours automated activity |
| Electronics merchant + large amount | Goods conversion / layering |
| Age < 25 + very large amount (≥p95) | Recruited mule indicator |
| Near-threshold structuring (within 1% of ceiling) | Refined threshold avoidance |

### Results

| Metric | Before (random labels) | After (rule labels) |
|---|---|---|
| ROC-AUC | 0.502 | **0.877** |
| Average Precision | 0.052 | **0.493** |
| Lift over baseline | 1.0× | **9.0×** |

Selected rule validation (fraud rate vs 5.5% baseline):

| Rule | Fraud rate | Lift |
|---|---|---|
| Low balance + large gift | 68.4% | 12.4× |
| Night + Transfer + large | 62.0% | 11.3× |
| Young donor + very large | 35.6% | 6.5× |
| High-risk platform + large | 29.0% | 5.3× |
| Amt/Balance ratio > p90 | 28.3% | 5.1× |

## Pipeline

### `generate_labels.py`

Reads the raw CSV, scores every row across 12 compliance rules, adds calibrated noise, applies a threshold to achieve ~5.5% anomaly rate, and writes the labeled CSV. Prints a validation table on exit showing fraud rate and lift for each rule.

### `prepare.py`

Trains three models and writes all artifacts to `artifacts/`:

| Model | Config |
|---|---|
| **Random Forest** | 300 trees, `class_weight='balanced'`, `max_depth=16`, `min_samples_leaf=4` — primary model |
| **Logistic Regression** | `class_weight='balanced'`, `max_iter=1000` — interpretable baseline |
| **Isolation Forest** | `contamination=0.055`, 200 trees — unsupervised, no labels needed |

Also writes `risk_df.parquet` (scored test set with risk tiers), `eda_cache.parquet` (lightweight EDA dataframe), `metrics.json`, and `meta.json`.

### `app.py`

Streamlit dashboard. All computation is pre-cached — the app only loads artifacts and renders plots.

## Features

| Feature | Description |
|---|---|
| `Log_Donation_Amount` | log1p of donation amount (reduces skew) |
| `Log_Cumulative_Given` | log1p of account balance |
| `Amt_Bal_Ratio` | Donation / (Balance + 1) — pass-through signal |
| `Amount_Zscore` | Z-score relative to full dataset |
| `Is_Night_Donation` | 1 if hour between 00:00–04:59 |
| `Is_Weekend` | 1 if Saturday or Sunday |
| `Is_Business_Hours` | 1 if hour between 09:00–17:00 |
| `Is_Round_Amount` | 1 if amount % 500 = 0 (structuring signal) |
| `Near_Threshold` | 1 if amount within 1% below 50k / 75k / 90k / 99k |
| `High_Risk_Platform` | 1 if Virtual Card, QR Code Scanner, or Banking Chatbot |
| `Is_Transfer` / `Is_Withdrawal` | Transaction type binary flags |
| `Is_Business_Account` | 1 if account type is Business |
| `Low_Balance_Large_Gift` | 1 if balance < p10 AND amount ≥ p75 |
| `Young_Large` | 1 if age < 25 AND amount ≥ p95 |
| Encoded categoricals | Gender, Region, Donation_Channel, Donation_Type, Campaign_Category, Device_Type, Currency |

## Dashboard

**Overview** — KPIs: total donations, anomaly count and rate, ROC-AUC, avg precision, night anomaly rate, high-risk platform count.

**Model Performance** — confusion matrix and precision-recall curves for all three models. Active model selected from the sidebar. Threshold adjustable via slider.

**Exploratory Analysis** — five tabs:
- *Amounts* — distribution histogram and boxplot by label
- *Temporal* — anomaly rate by hour of day and day of week
- *Geography* — regional anomaly rates (top 20) with donation count overlay
- *Categories* — anomaly rate by donation type, campaign category, channel, device, currency
- *Features* — Random Forest feature importance

**Risk Triage Queue** — sortable, filterable table of donations scored Low / Medium / High / Critical with progress bar scores.

**Single Donation Scorer** — enter donation details and get an instant anomaly score, risk tier, and list of which compliance rules fired — useful for explaining flags to reviewers.

## Ethical Notes

> The labels in this system are synthetically generated from domain rules, not from real compliance review outcomes. In production, all Critical and High flags should go through human review before any action is taken against a donor.

- **Human-in-the-loop** — Critical/High flags should require compliance officer sign-off before a donor is blocked or contacted
- **Appeals mechanism** — donors incorrectly flagged need a clear process to dispute
- **Bias audits** — regularly test whether any demographic group is being systematically over-flagged
- **Sanctions integration** — add OFAC/UN sanctions API as a hard-block layer before the ML score
- **Feedback loop** — compliance review outcomes should feed back into retraining monthly
- **SHAP explanations** — add SHAP values per donation for explainable audit trails, required for regulatory reporting in most jurisdictions
- **Real labels** — retroactively label historical suspicious donations with compliance team input to replace synthetic labels over time
